## Overview

This project is a production-ready computer-vision pipeline that detects, tracks, and measures the real-time speed of every vehicle visible in a highway traffic video — without requiring any infrastructure changes to the road.

The system uses **YOLOv8** for frame-by-frame vehicle detection, **ByteTrack** for stable multi-object tracking, and a **Perspective Homography Matrix** to convert skewed 2-D pixel coordinates into a metrically accurate flat-ground coordinate system where distances correspond directly to real-world metres.

---

## Why Homography Instead of Virtual Timing Lines?

Earlier approaches to camera-based speed measurement rely on two virtual lines drawn across the video frame. A vehicle's speed is computed from the time it takes to cross from line 1 to line 2.

This method has two fundamental limitations:

### 1 — Perspective Compression

A traffic camera mounted at an angle creates a non-linear pixel-to-metre mapping. Vehicles near the **top** of the frame (further from the camera) appear much smaller than vehicles near the **bottom** (closer). One pixel of vertical motion near the top of the frame represents far more real-world distance than one pixel near the bottom.

A two-line system partially compensates for this by placing both lines at a fixed location, but any measurement made **between** those lines — or outside them — is distorted. The error is proportional to how far the vehicle deviates from the line region.

### 2 — Discrete Measurement

A vehicle's speed is computed only at the moment it crosses both lines. Between measurements, the displayed value is frozen — giving no insight into acceleration, deceleration, or continuous behaviour.

---

### The Homography Solution

`cv2.getPerspectiveTransform()` solves a system of 8 linear equations to find the unique **3×3 projective matrix M** such that the four corners of a known road quadrilateral map exactly to a flat, metrically correct rectangle:

```
[u']     [m00  m01  m02]   [u]
[v']  ~  [m10  m11  m12] × [v]
[w']     [m20  m21  m22]   [1]

world_x = u' / w'
world_y = v' / w'
```

After this transform, **equal differences in warped coordinates correspond to equal real-world distances everywhere in the scene** — regardless of where in the frame the vehicle is. The system can then compute a fresh, perspective-correct speed estimate for every single frame, which is smoothed over a configurable sliding window to suppress bounding-box jitter.

## Architecture

```
main.py
  └── TrafficAnalyticsEngine (src/pipeline.py)
        │
        ├── PerspectiveTransformer (src/homography.py)
        │     • Builds 3×3 homography matrix once at startup
        │     • pixel_to_world(px, py) → (x_m, y_m)
        │
        ├── VehicleTracker (src/tracker.py)
        │     • Registry of VehicleState objects keyed by ByteTrack ID
        │     • Creates new states on first detection
        │
        │     VehicleState (src/tracker.py)
        │           • Sliding-window speed buffer (deque)
        │           • Direction estimator (sign of mean Δy)
        │           • smooth_speed, direction, label helpers
        │
        └── YOLO + ByteTrack (ultralytics)
              • model.track() returns bounding boxes + stable track IDs
```

**Per-frame data flow:**

```
Read frame → Resize to 1280×720 → YOLO.track()
  │
  └── for each detection:
        foot_point = (bottom_centre_x, y2)
        world_pos  = PerspectiveTransformer.pixel_to_world(foot_point)
        state      = VehicleTracker.update(track_id, world_pos, timestamp)
        Draw: green bounding box + ID label + speed label
```

## Installation

### Prerequisites

- Python 3.9 or later
- A CUDA-capable GPU is optional but significantly improves throughput.
  CPU inference works but may run below real-time on 4K input.

### Steps

```bash
# 1. Clone the repository
git clone https://github.com/YOUR_USERNAME/traffic-analytics-engine.git
cd traffic-analytics-engine

# 2. (Recommended) Create a virtual environment
python -m venv .venv
# Windows:
.venv\Scripts\activate
# macOS / Linux:
source .venv/bin/activate

# 3. Install dependencies
pip install -r requirements.txt
```

YOLOv8 model weights (`yolov8n.pt`) are downloaded automatically by
Ultralytics on the first run.

## How to Calibrate — Setting SRC_POINTS

Calibration maps four pixel points on your camera frame to their known
real-world positions. Getting this right is the single most important step
for accurate speed readings.

### Step 1 — Extract a reference frame

```bash
python tools/extract_frame.py
```

This saves `assets/frame0.png` at 1280×720 resolution.

### Step 2 — Pick four road-surface corners interactively

```bash
python tools/extract_frame.py --pick
```

An OpenCV window opens. **Click four points** on the road surface in this
exact order:

| Click | Corner | Description |
|-------|--------|-------------|
| 1st | **TL** — Top-Left | Far-left corner of your road patch |
| 2nd | **TR** — Top-Right | Far-right corner of your road patch |
| 3rd | **BR** — Bottom-Right | Near-right corner of your road patch |
| 4th | **BL** — Bottom-Left | Near-left corner of your road patch |

The script prints the pixel coordinates and generates ready-to-paste Python
code for `config.py`. Press **Enter** or **Q** to close.

```
── Paste this into config.py ──────────────────────────────
SRC_POINTS = np.float32([
    [ 490,  310],   # TL
    [ 790,  310],   # TR
    [ 980,  600],   # BR
    [ 300,  600],   # BL
])
────────────────────────────────────────────────────────────
```

### Step 3 — Measure the real-world dimensions

Open **Google Maps** (satellite view) and use the distance ruler to measure:

- `REAL_WIDTH_M` — the physical distance from TL to TR on the ground (metres).
- `REAL_LENGTH_M` — the physical distance from TL to BL on the ground (metres).

A typical two-lane highway section spanning ~30–40 metres of depth is ideal.

```python
# config.py
REAL_WIDTH_M  = 8.0    # e.g. two 4-metre lanes
REAL_LENGTH_M = 35.0   # depth of road patch covered
```

### Step 4 — Validate

Run the engine and observe vehicles whose approximate speed you can estimate
(e.g. highway cruise at ~110 km/h). Adjust `REAL_LENGTH_M` iteratively:

| Symptom | Fix |
|---------|-----|
| All speeds read too high | Decrease `REAL_LENGTH_M` |
| All speeds read too low | Increase `REAL_LENGTH_M` |
| Speeds vary wildly | Increase `SPEED_BUFFER_LEN` in config.py |
| Vehicles at far end read differently from near end | SRC_POINTS quad is skewed — re-pick the corners more carefully |

## Configuration Reference

All parameters are in `config.py`. Key values:

| Parameter | Default | Description |
|-----------|---------|-------------|
| `VIDEO_PATH` | *(your path)* | Source video file or RTSP URL |
| `DISPLAY_WIDTH / HEIGHT` | 1280 × 720 | Processing resolution |
| `SRC_POINTS` | *(example quad)* | Four pixel corners of road patch |
| `REAL_WIDTH_M` | 10.0 | Road patch width in metres |
| `REAL_LENGTH_M` | 40.0 | Road patch length in metres |
| `MODEL_NAME` | `yolov8n.pt` | YOLO model weights |
| `CONF_THRESHOLD` | 0.35 | Minimum detection confidence |
| `SPEED_BUFFER_LEN` | 8 | Moving-average window (frames) |
| `MAX_PLAUSIBLE_SPEED_KMH` | 250.0 | Hard clamp on impossible speeds |
| `VEHICLE_CLASSES` | [2, 3, 5, 7] | COCO IDs: car, moto, bus, truck |

---

## Performance Tips

| Goal | Action |
|------|--------|
| Faster inference | Use `yolov8n.pt` (default) or reduce `DISPLAY_WIDTH/HEIGHT` |
| Higher accuracy | Switch to `yolov8s.pt` or `yolov8m.pt` |
| GPU acceleration | Install PyTorch with CUDA; Ultralytics detects it automatically |
| Smoother speed readout | Increase `SPEED_BUFFER_LEN` (try 12–15) |
| Reduce false positives | Raise `CONF_THRESHOLD` (try 0.45–0.55) |
